# Project Status & DB Migration Plan

This doc is a snapshot of what has been built and exactly what changes when
the real database replaces the temporary local SQLite one.

---

## 1. What is done so far

### Architecture (3 clean layers, decoupled)

```
User text
   │
   ▼
detect_intent()            ── regex keyword classifier (Phase 2)
   │
   ▼
INTENT_TO_SQL[intent]      ── parameterized SELECT queries (Phase 2)
   │
   ▼
format_response(...)       ── rule-based healthcare advice (Phase 3)
```

Plus `explain_alert(hospital_id)` — a multi-step "Why is this hospital at
risk?" handler that joins HospitalsTable + InventoryTable and computes
weeks-of-supply.

### Files in the repo

| File                 | Status           | Purpose                                          |
|----------------------|------------------|--------------------------------------------------|
| `schema.sql`         | **temporary**    | SQLite 3NF CREATE TABLE + seed data              |
| `init_db.py`         | **temporary**    | Rebuilds `healthcare.db` from `schema.sql`       |
| `healthcare.db`      | **temporary**    | Local SQLite file generated by `init_db.py`      |
| `chatbot_backend.py` | keep             | Intent classifier, query map, formatter, router  |
| `app.py`             | keep             | Streamlit chat UI                                |
| `requirements.txt`   | keep (extend)    | Python dependencies                              |
| `README.md`          | keep             | Setup, intent map, test cases                    |

### Intents implemented

| Intent ID                 | Triggers                                             | SQL target                                   |
|---------------------------|------------------------------------------------------|----------------------------------------------|
| `EXPLAIN_ALERT`           | "why …", "why is Hospital X at risk"                 | `HospitalsTable` + `InventoryTable` join     |
| `CHECK_INVENTORY`         | "low", "shortage", "stock", "inventory", "how much"  | `InventoryTable` (optionally by blood type)  |
| `GET_HIGH_RISK_PATIENTS`  | "priority", "risk", "surgery", "urgent patients"     | `PatientsTable` where `RiskScore > 7`        |
| `GET_TRANSPLANT_PRIORITY` | "transplant", "waiting list", "matching"             | `OrganRequests` ⋈ `PatientsTable`            |
| `GET_DONORS`              | "donor", "eligible"                                  | `DonorsTable` where `EligibilityStatus='Eligible'` |
| _fallback_                | anything else                                        | returns the fixed fallback sentence          |

### Rule-based "intelligence" layered on top

| Rule                                   | Behavior                                                               |
|----------------------------------------|------------------------------------------------------------------------|
| `CurrentUnits < 5`                     | append `URGENT: Stock is critically low.`                              |
| `5 ≤ CurrentUnits < 10`                | tag row as `low` + message "Please plan a shipment soon."              |
| `CurrentUnits ≥ 10`                    | tag `sufficient`                                                       |
| `RiskScore > 8`                        | patient tagged `High Priority`                                         |
| `HemoglobinLevel < 7`                  | patient tagged `Critical`                                              |
| `7 ≤ HemoglobinLevel ≤ 10`             | patient tagged `Urgent`                                                |
| `HemoglobinLevel > 10`                 | patient tagged `Moderate`                                              |
| `CurrentStock < AverageWeeklyUsage`    | hospital flagged AT RISK with the exact "Why" sentence                 |
| Empty result set                       | returns `I could not find any relevant data for that request.`         |

### Safety guarantees (already enforced)

- **SQL-injection safe** — every user-derived value is passed via `?`
  placeholders, never string-formatted into SQL.
- **Read-only** — SQLite connection opened with `mode=ro`; only hard-coded
  `SELECT` statements live in the code; a `_assert_select_only()` guard
  rejects anything else.
- **Output bounded** — every listing query uses `LIMIT 10`.
- **No hallucinations** — unmatched input returns a fixed fallback, empty
  results return a fixed "no data" sentence.

## 3. What will be replaced when the real database is ready

### 3.1 Will be deleted / retired

| File            | Why                                                                      |
|-----------------|--------------------------------------------------------------------------|
| `schema.sql`    | Real DB will be managed by the DBA / a separate migration tool           |
| `init_db.py`    | Not needed — the real DB is already populated                            |
| `healthcare.db` | Local SQLite file no longer used                                         |

### 3.2 Will change inside `chatbot_backend.py`

All of these live in **one file**; the rest of the architecture stays.

1. **Connection layer** — `_connect_readonly()` and the `DB_PATH` constant.

   Currently:

   ```python
   DB_PATH = "healthcare.db"

   def _connect_readonly(db_path: str) -> sqlite3.Connection:
       return sqlite3.connect(f"file:{db_path}?mode=ro", uri=True)
   ```

   After migration (example for MySQL):

   ```python
   import mysql.connector
   import os

   def _connect_readonly():
       return mysql.connector.connect(
           host=os.environ["DB_HOST"],
           port=int(os.environ.get("DB_PORT", 3306)),
           user=os.environ["DB_READONLY_USER"],   # account with SELECT-only grants
           password=os.environ["DB_READONLY_PASS"],
           database=os.environ["DB_NAME"],
       )
   ```

2. **Parameter placeholder syntax** — SQLite uses `?`, MySQL and Postgres use
   `%s`. Every entry in `INTENT_TO_SQL` and `EXPLAIN_QUERIES` has `?`
   placeholders. If you move to MySQL/Postgres, do a single find-and-replace
   from `?` to `%s`, or keep `?` and wrap with a tiny helper.

3. **Table / column names** — if the real DB uses different casing or naming
   (e.g. `HOSPITALS`, `blood_inventory`), update the SQL strings in
   `INTENT_TO_SQL` and `EXPLAIN_QUERIES`. Nothing outside those two dicts
   references tables directly.

4. **Read-only enforcement** — SQLite `mode=ro` goes away. Replace it with a
   **DB-level grant**: create a user that only has `SELECT` on the relevant
   tables/views. The `_assert_select_only()` guard in the code is kept as
   defense-in-depth.

5. **Data types** — `SurgeryScheduled` is stored as `0/1` today; in MySQL it
   might be `BOOLEAN` / `TINYINT(1)`. The formatter only checks truthiness,
   so no code change is needed, but verify during testing.

### 3.3 Will NOT change

These stay exactly as they are:

- `INTENT_PATTERNS` (regex classifier)
- `format_response` and all `_format_*` helpers
- `_categorize_hemoglobin`, `_stock_tag`
- Threshold constants (`URGENT_STOCK_THRESHOLD`, `LOW_STOCK_THRESHOLD`,
  `RISK_THRESHOLD`, `HIGH_PRIORITY_RISK`, `HB_CRITICAL`, `HB_URGENT`)
- `process_user_query` router
- `app.py` (Streamlit UI) — it only imports `process_user_query`

### 3.4 Migration checklist

- [ ] Add real DB driver to `requirements.txt`
      (`mysql-connector-python` / `psycopg2-binary` / etc.)
- [ ] Create a **read-only DB user** and grant `SELECT` on the 5 tables
- [ ] Put connection details in environment variables (not in code)
- [ ] Replace `_connect_readonly()` with the driver-specific version
- [ ] Update placeholder syntax if the driver uses `%s` instead of `?`
- [ ] Run `INTENT_TO_SQL` queries against the real DB manually to verify
      column names match; adjust SQL strings if needed
- [ ] Delete `schema.sql`, `init_db.py`, and `healthcare.db`
- [ ] Re-run the smoke tests from README.md against the real DB
